# fleet-audit-loop

[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx) Generated by Claude under human direction. The rotation model, the dimension roster, and the briefing rules were defined by a human from eleven rounds of production audit evidence.

An audit that re-runs the same sweep until the findings taper has not converged; it has run out of places to look. This skill loops a fleet of auditors over a set of agent or skill definitions so that each round points a different auditor at a different dimension, and terminates when the dimensions are exhausted rather than when a round comes back quiet.

## What it does

Runs a multi-round audit over agent and skill definitions. Each round selects a dimension from a fixed roster, dispatches the auditor that covers it, and reads the result from a report file on disk rather than from the agent's reply. Findings are refuted before they are fixed, dispositioned in a ledger, and where a finding expresses an invariant that can fail, the fix step writes a check as well as the fix.

## Key ideas

- **Rotate dimensions, do not repeat the sweep.** Across eleven rounds, findings tracked wherever each round was pointed and did not decay with repetition: rounds aimed at a new dimension were still producing CRITICALs and HIGHs after eight rounds. A falling finding count across rounds that reuse a dimension means the sweep is exhausted, not the artefact.
- **Termination is on dimensions exhausted, not on a clean round.** A clean round in one dimension says nothing about the next. The honest limit is stated in the output: six dimensions is what this fleet currently has, and running out of them is not proof the artefact is clean.
- **Brief with classes, not instances.** Each round carries recurring defect *patterns* from prior rounds, never the specific defects already found and fixed. Pattern briefing produced findings in new locations; instance briefing produced confirmation of the instances. Round 6 is the deliberate exception, because confirming a named list of applied fixes is what `fix-regression-checker` is for.
- **The report file is what the loop reads.** An agent gone quiet with `Status: IN PROGRESS` died mid-pass; one gone quiet with no file never started. Neither is done and neither reports itself, so every round is verified on disk before it counts. Re-prompting is the fallback, not the mechanism.
- **Nothing is dispatched onto work that is merely idle.** Before a round opens, every report file from the previous round must read `Status: COMPLETE`. Dispatching a second agent onto a file another was still holding caused a near-collision caught by luck rather than by a check.
- **Prefer a gate to prose where an invariant can fail.** Parallel-place omission was the single most recurrent defect, eleven occurrences across eleven rounds, including one by an agent explicitly briefed about that exact pattern. Briefing did not prevent it. Where a finding can be expressed as a check that fails, the fix writes the check too - and a check that cannot fail is worse than none, because it is read as coverage.
- **The rules reach the fleet only through the brief.** No agent in the fleet declares `Skill` among its tools, so a dispatched auditor cannot read this skill. Sections meant for an auditor are marked for relay into the dispatch brief, which is the only route available.

## Scope boundary

`fleet-qa-loop` governs code, running mechanical linters before the QA fleet. `plan-audit-loop` governs plans, looping `plan-auditor` and `requirements-auditor` over a document before execution begins; this skill neither supersedes nor wraps it. This skill governs definitions: the agent and skill files that are themselves prompts.

## Status

The mechanism is unrun as of 2026-07-18. It is written from eleven rounds of audit evidence, but the loop has not been executed end to end, so two properties remain untested: that a round's brief in practice carries the previous round's patterns rather than its instances, and that the loop reads the previous round's report file rather than falling back to re-prompting. The skill says so in its own body rather than describing an unrun design as verified.
