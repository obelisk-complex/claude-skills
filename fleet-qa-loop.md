# fleet-qa-loop

[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx) Generated by Claude under human direction. The loop structure, ordering rules, and reporting discipline were defined by a human and refined through production use.

A Claude skill that QAs any target until it is genuinely clean: mechanical linters first, then the qa/audit subagent fleet, looping until lint is clean, tests are green, and every dispatched agent returns a PASS verdict.

## What it does

Runs a loop-until-clean quality pass over any codebase or target. Cheap mechanical linters run first and auto-fix what they safely can; the judgment-level work is then handed to the audit/qa subagent fleet. The loop re-runs lint and tests and re-dispatches agents until the target is clean or progress stalls - so a "qa pass" converges instead of being a one-shot opinion.

## Key ideas

- **Mechanical before fleet.** Linters (`ruff`, `eslint`, `golangci-lint`, `clippy`, `shellcheck`, ...) catch missing imports and formatting for free; agents are reserved for judgment-level findings.
- **Generalised to any target.** Detects the toolchain from the files present, verifies each tool exists before invoking it, and skips-with-a-note when one is absent.
- **Loop til clean, but never spin.** Terminates on CLEAN (lint green, tests green, every agent PASS) or STALLED (a finding survives two rounds, an agent returns BLOCKED, or an iteration cap is hit).
- **Append-as-you-go reporting.** Every dispatched agent writes its report skeleton first and appends each finding the moment it is found, so a truncated or dead agent still leaves its findings on disk - an incomplete report beats a complete report lost.
- **Adversarial verification before any fix.** A finding graded only by the agent that produced it is a self-assessment. Each finding faces three independent refuters - briefed to *kill* it, given the code but not the author's reasoning, and told to default to "refuted" when uncertain. Only findings that survive get fixed; refuted ones are recorded rather than silently dropped, so the fleet's false-positive rate stays visible.
- **The tree's own claims are not evidence.** A comment asserting what some external system does - a compositor, a GPU driver, a kernel, a version of another program - is checked by nothing and covered by no test, yet it reads as hard-won field knowledge and the next reader will reason from it. Such claims are treated as findings, not facts: check what hardware and software the claim needs, then check whether anyone had it. `scripts/check-claims.sh` flags added comments that assert external behaviour without citing a spec, man page, or URL; marking a claim `UNVERIFIED` clears it, so a possibly-true observation survives without a guess masquerading as a measurement.
- **Chesterton's fence on "unused" code.** Never blind-deletes a flagged-unused variable: an unused computed value is often a missing-use bug, not dead code.
