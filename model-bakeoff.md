# model-bakeoff

[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx) Generated by Claude under human direction. The five failure modes and the worked numbers were drawn from a human's own model evaluations and refined through production use.

Comparing candidates against ground truth to pick a winner is not primarily a modelling problem. The danger isn't picking the wrong model, it's producing a number that looks like evidence and isn't, then spending real money on it. Most of what kills a bake-off is the harness, the sample, and the arithmetic, not the model.

## What it does

Domain: model and prompt evaluation methodology. Reader: whoever is benchmarking extraction, classification, or OCR candidates, comparing an incumbent against alternatives, or about to act on numbers a bake-off has produced.

It assumes the basics are already in place: ground truth built by hand, recall measured alongside precision, the incumbent's own output never trusted as truth. Baseline testing showed agents reliably get those right. What this skill covers are five specific things they reliably get wrong when scoring a comparison, plus a short list of aggregate-hiding failure modes worth naming in the report rather than folding into one accuracy figure.

The discipline underneath all five: run each candidate at least twice, and keep the runner (which spends money and saves raw output to disk) separate from the scorer (which is free, offline, and rerunnable). You will re-score more often than you expect, and should never pay twice for the same read.

## The five failure modes

1. **A failed request is not a model error.** Rate limits, timeouts, an upstream returning no choices: these are facts about infrastructure, not the model. Retry failures before comparing anything, and report them separately from accuracy. One 429 storm cost a candidate 7 of 46 pages and read as 74% recall; retried, it was fine, and that number would have eliminated the best model in the field.
2. **Failures must stay local and visible.** Catch parse errors the way you catch API errors: per item, recorded, continue. Set a per-request timeout so one hung candidate can't silently block every candidate behind it. Never silently drop a failed item, a vanished item can't be counted as missed, so recall quietly flatters the model.
3. **Compute the do-nothing baseline from the current test set, at run time, never hard-coded.** Print it beside every score, and report accuracy on the non-majority items separately, that's the number that says whether a model can read rather than guess. One incumbent that had processed 40,000 rows scored 65.1% against a 63.4% baseline nobody had ever computed.
4. **Measure cost from real tokens, don't extrapolate it.** Run a handful of real items and price from actual input and output tokens; one sample item isn't a measurement, since items can vary 3x. A cost estimated from a single probe page said $47; measured across 46 real pages, it was $70.
5. **One run is not a measurement.** Run each candidate at least twice on the same items; disagreement between a model and itself bounds how much you can trust its margin over another.

## What to report separately

Aggregate accuracy hides the errors that cost the most money:

- **Correlated / whole-item collapse**: every row on a page sharing one wrong value inverts the item, it isn't noise. Count collapsed items, not just wrong rows.
- **Fabrication rate**: a confident wrong value in-distribution is the only error that spends money. State it separately from accuracy.
- **Fields the schema never asked for**: if the output schema omits a flag, every model looks equally bad at it. That's a schema bug, not a model weakness.
- **Values normalised before comparison**: `2/19/2026` versus `02/19/2026` compared as strings is a fake error rate. Parse, then compare.
