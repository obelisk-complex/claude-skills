# readme-conventions

[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx) Generated by Claude under human direction. The structure, the benefit-first ordering, the disclosure rules and the badge requirement were defined by a human and refined across their own repos.

Domain: README authoring for this operator's repos, most of them private, single-operator, and holding real data. Reader: the repo owner, plus any collaborator he brings in later. He does not need his own stack explained back to him.

Use when writing or rewriting a README, or when a release adds capabilities the README does not yet describe.

## The shape

In this order. The two collapsed blocks are load-bearing: they keep the top of the file about what the reader gains, and push the mechanics down to where someone looking for mechanics will go.

1. **Title and one line** on what the repo is.
2. **Quillx badge**, immediately after, on its own line:
   `[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx)` followed by one or two sentences describing the actual division of labour.
3. **Screenshots**, collapsed in a `<details>` block with a `<summary>` label. Relative paths.
4. **What this gives you.** The heart of the file. Per capability: what the operator can now do, then how. Not a feature list.
5. **Setup and running.** Enough to rebuild on a fresh machine.
6. **Security posture**, where the project handles real data. Brief and factual, sourced from the repo's own threat model.
7. **Technical changelog**, collapsed in a `<details>` block, last. What changed, in technical terms: commits, files, mechanisms.

## Benefit before mechanism

The rule that makes the difference: every capability leads with what it gets you, and only then how it works.

- **Avoid:** "Adds `cmd/analytics-sync`, a cron-driven drain loop over the export endpoint."
- **Prefer:** "See who is hitting the site, and whether it is an attack. A poller drains the collector every 15 minutes into a local mirror."

Mechanism is not deleted, it moves. Anything a reader would need only when changing the code belongs in the collapsed changelog.

## The badge is a factual claim

The Quillx line describes who did what. It is the one line in the README a reader has no way to verify, so it carries the weight of your accuracy elsewhere.

Write what actually happened. If a human defined the threat model, chose the architecture, and made the security calls while Claude wrote the implementation and tests, say that. If a human wrote the whole thing and Claude only edited prose, say that instead. Inflating either side defeats the purpose of disclosing at all.

Level 4 (`quillx-4.svg`) is this operator's usual level and means "generated under human direction". Use a different level when the true division of labour differs, not to be modest or generous.

## Screenshots

Include them where the project has a UI worth seeing.

**Where the project touches real, personal or customer data, screenshots show synthetic or fixture data. Not "preferably" - always.** A screenshot of a real dashboard puts real identifiers into git history, and a private repo is not a safeguard: it can be shared, forked, made public, or backed up somewhere less careful later. Say in the block that the data is fixture data, so nobody mistakes it for a live view.

Before saving any screenshot, look at the image and confirm what is actually in the frame. Browser chrome catches bookmark bars, other tabs, notifications and window titles. Capture the content area rather than the whole desktop.

A screenshot that shows a page mid-scroll or cut off reads as broken even when the page is fine. Either capture the whole view or accept the crop deliberately.

## House style

- **No em-dashes.** Use `-`, commas, parentheses, colons, or separate sentences.
- **British English.**
- No invented licence, CI badge, contribution guidelines, roadmap, or support channel. If it does not exist, it does not go in.
- Private single-operator repos get no "contributions welcome" boilerplate.
- Match the disclosure level already set by committed files. If `.env.example` commits an endpoint, the README may name it; if a value is secret there, it stays secret here.

## Producing one

**Step 1: read the real source.** Every factual claim comes from a file you have opened: entrypoints, `Makefile`, `.env.example`, cron and deploy config, the existing README, the threat model, the runbooks. A documentation task on one of these repos previously invented an architecture when it was given a plan summary instead of code, so paraphrases and second-hand descriptions are not a sufficient basis.

**Step 2: draft**, in the shape above.

**Step 3: verify.** List the factual claims in the draft - paths, command names, env var names, behaviours, version numbers. Check each against the file it came from, working from the files rather than re-reading the draft for plausibility. Correct what does not match.

This step regularly finds real bugs rather than doc errors: env var names the code does not read, make targets that no longer exist, documented flags with no implementation. When a claim in the docs and the behaviour in the code disagree, that is a finding about the project, not just about the README. Report it rather than quietly writing around it.

**Step 4: run the prose through the `copywriting` skill's AI-tell gate.** That skill holds the tell blacklist and the verification pass.

If you cannot verify a claim, leave it out or mark it unverified. An accurate short README beats a comprehensive one carrying invented detail, and "I could not confirm this" is a useful result.
