# readme-conventions

[![Quillx](https://raw.githubusercontent.com/qainsights/Quillx/main/badges/quillx-4.svg)](https://github.com/qainsights/Quillx) Generated by Claude under human direction. The structure, the benefit-first ordering, the disclosure rules, and the badge requirement were defined by a human and refined across their own repos.

The house convention for writing or rewriting a README: section order, the benefit-before-mechanism rule, disclosure blocks, badge honesty, and screenshot policy, calibrated for one operator's own repos rather than a generic open-source template.

## What it does

Domain: README authoring for this operator's repos, most of them private, single-operator, and holding real data. Reader: the repo owner, plus any collaborator he brings in later, who does not need his own stack explained back to him.

The shape has two collapsed `<details>` blocks doing load-bearing work: screenshots near the top and a technical changelog at the bottom, both folded so the visible top of the file stays about what the reader gains, not how it works. Everything else follows the same rule at the sentence level: every capability leads with what it gets you, then how. "See who is hitting the site, and whether it's an attack. A poller drains the collector every 15 minutes into a local mirror" beats "Adds `cmd/analytics-sync`, a cron-driven drain loop over the export endpoint." Mechanism isn't deleted, it moves to the changelog.

Producing one is a four-step, source-first process: read the real entrypoints, `Makefile`, `.env.example`, and existing README rather than a plan summary; draft in the shape below; verify every factual claim (paths, command names, env var names, version numbers) against the file it came from; then run the prose through the `copywriting` skill's AI-tell gate. Verification regularly surfaces real bugs, an env var the code doesn't read, a make target that no longer exists, rather than doc errors, and those get reported as findings about the project, not quietly written around.

## The shape

In order: title and one line on what the repo is; the Quillx badge with one or two sentences on the actual division of labour; screenshots collapsed in a `<details>` block, relative paths; "What this gives you", the heart of the file, per capability, benefit then mechanism; setup and running, enough to rebuild on a fresh machine; a brief, factual security posture where the project handles real data; and a collapsed technical changelog last.

## House rules

- The badge is a factual claim, the one line a reader has no way to verify, so it carries the weight of everything else's accuracy. Level 4 means "generated under human direction" and is this operator's usual level; use a different level when the true division of labour differs, never to be modest or generous.
- Where a project touches real, personal, or customer data, screenshots always show synthetic or fixture data, said explicitly in the block, because a private repo is not a safeguard against it later being shared, forked, or made public.
- No em-dashes; British English; no invented licence, CI badge, contribution guidelines, roadmap, or support channel. Private single-operator repos get no "contributions welcome" boilerplate.
- Disclosure matches what committed files already reveal: if `.env.example` names an endpoint, the README may too; if a value is secret there, it stays secret here.
- An accurate short README beats a comprehensive one carrying invented detail. Where a claim can't be verified, it's left out or marked unverified.
